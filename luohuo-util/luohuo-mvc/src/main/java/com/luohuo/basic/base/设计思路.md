# 设计思路：五层调用模型

本系统借鉴了阿里的规范，采用 5 层调用模型：`controller -> biz -> service -> manager -> mapper`，其中 `biz` 层是可选的。

## controller

1. 请求转发

## biz（可选）

1. 处理不同数据源的业务逻辑。
2. 可能存在分布式事务（不能使用本地事务，即该层方法或类上不能加 `@Transactional`）
3. 当某个业务仅仅只会操作一个具体的数据源时，可以不要biz层，直接使用service层

## service

1. 控制事务
2. 处理同一数据源的业务逻辑

## 总结

1. `service` 的 `save` 方法更贴切实际业务；`manager` 的 `save` 方法只负责单表保存（可对字段设置默认值）；`mapper` 的 `insert` 方法只负责原封不动插入数据
2. 对于复杂的业务，避免service互相依赖
3. 调用只能从上往下，不能反着调用，最好也不要平层交叉调用。
4. 尽量避免平层交叉调用
